Memory Deep Dive

Why File-Based Memory Is Secretly Genius

Most AI assistants have one type of memory: the conversation window. Once a conversation gets long enough, the oldest messages fall off the edge. Details are gone forever.

Barak's memory works differently. The file system is the memory — and it has no upper limit. Every fact, preference, spec, and lesson learned gets written to markdown files on disk. Those files don't expire, don't get compressed, and don't fall off any edge.

The insight: this turns memory into a search problem, not a capacity problem. And search is essentially a solved problem that keeps getting more efficient. As OpenClaw develops, the tools for searching and recalling from this file system are getting sharper every week. The vision is that the file system becomes virtually indistinguishable from a limitless context window.

In theory, this means infinite memory with perfect recall. In practice, we're not there yet.

⚠️ Where It Breaks Down Today — Honestly

The write problem.

Memory is only as good as what gets written down. If you tell Barak something in passing and he doesn't save it to a file, future sessions won't know. Saying "save that" helps force it.

The staleness problem.

Files can become outdated. If a preference changes but the old file isn't updated, Barak will act on stale information. There's no automatic expiry on a markdown file.

The search-not-read problem.

Barak searches memory files before responding, but search depends on asking the right question. If a fact is filed under a heading that doesn't match, he might miss it.

The verify-before-asserting problem.

The biggest failure mode: Barak thinking he remembers something when he's actually confabulating from compressed summaries. The fix: always check the file before asserting the fact.

🔧 What's Improving

OpenClaw is under active development:

  • Better automatic writes — detecting important facts and persisting them without being told
  • Smarter search — semantic recall that works more like natural questions
  • Lossless context compression — summaries that preserve critical details
  • File system as context — treating the entire workspace as an extension of the context window

The architecture is right. The execution is catching up.

Identity Files — Who Barak Is

Four files define Barak's identity. They're loaded into his system prompt at the start of every session. Think of them as his DNA: they shape how he thinks, talks, and acts.

⚡
SOUL.md
Personality & behavior rules

The deepest layer. Defines personality, communication style, and boundaries. Where "sharp and direct, no filler" comes from.

  • Controls: Tone, personality, values, boundaries
🪪
IDENTITY.md
Name, model, basic facts

The basics: name, emoji, model. Simple facts that stay constant. A birth certificate.

  • Controls: Name, username, model, creation date
👥
USER.md
Who Barak works for

Everything about the people he serves — names, roles, timezones, preferences. Where "Menachem is in LA, signal over volume" lives.

  • Updated by: Telling Barak about yourself
⚙️
AGENTS.md
Operational configuration

The rulebook: repo access, git workflows, responsibilities, startup procedures.

  • Set by: Michael (admin-level config)

💬 How to Edit These: Just Talk

You don't need to edit files. Just tell Barak what you want:

# Valid ways to shape Barak's identity:

"Be more concise in briefings. Bullet points only."
"Don't use emojis in serious reports."
"I prefer you flag uncertainties explicitly."
"My timezone changed — I'm in NYC now."

# The richer, the better. Describe:
# → Communication style, decision-making approach
# → How to handle ambiguity
# → Specific phrases to use or avoid

→ Barak updates the file, takes effect next session.

⚠️ The Drift Problem

Identity files are loaded at session start. As conversations get longer, the context window fills up and identity instructions get proportionally smaller. Barak can gradually "drift" from his configured personality.

Session start~95%
Mid-session (~50k tokens)~60%
Long session (~150k tokens)~25%

If Barak starts feeling "off" — less sharp, more generic — it's probably drift. A fresh session resets it. Or just tell him: "You're drifting, re-read your SOUL."

Tips for Working With Barak

  • ✅
    Be explicit and say "write it down."
    "My timezone is PT, I'm in LA. Save that." — locks it into persistent memory.
  • ✅
    Call out mistakes directly.
    Barak can check logs, configs, and state — but only if he knows something's off.
  • ✅
    Think of requests as specs, not wishes.
    "Send me news at 6:30 AM PT, max 5 bullets, real journalism only" — that's a spec he can implement.
  • ✅
    Ask him to build things.
    Not just Q&A — recurring workflows, monitoring scripts, data pipelines.
  • ❌
    Don't assume he remembered something from weeks ago.
    If it wasn't written to a file, it may have faded. When in doubt, restate it.
  • ❌
    Don't let him get away with vague answers.
    If Barak says "it should be working" without proof — push back.
